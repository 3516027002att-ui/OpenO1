# OpenO1 的系统级推理定位与能力补偿假设

- 日期：2026-07-13
- 类型：项目定位 / 研究假设 / 决策记录
- 状态：草案

## 背景

本地开放权重模型仍在发展，但消费级 4B 到 9B 模型的纯文本复杂推理增速已经明显放缓。OpenO1 不应把项目成立建立在“小模型会持续快速接近前沿模型”这一单一前提上。

项目更稳定的定位，是研究如何利用 Agent 协作、结构化上下文、工具调用、独立验证和迭代修复，把基础模型的单次生成能力组织成更强的系统级问题解决能力。

## 核心定义

OpenO1 的目标可以表述为：

> 利用 Agent、上下文管理和工具调用，弥补基础模型在长程规划、状态保持、独立验证、外部计算和错误修复上的不足，将单次模型能力转化为可持续、可验证的系统推理能力。

这里的“弥补”存在边界。基础模型至少需要具备：

1. 基本理解任务的能力；
2. 遵循结构化协议的能力；
3. 根据具体反馈修正结果的能力。

当模型连任务和反馈都无法稳定理解时，多 Agent 会并行放大错误；当模型具备基本能力但容易漏条件、算错、遗忘状态或缺少外部知识时，框架增强最可能产生明显收益。

## 三根主要支柱

### Agent 协作

弥补单次生成的路线锁定和视角单一：

- 多路线探索；
- 独立验证；
- 对抗性找错；
- 根据 blocker 定向修复；
- 复杂任务的并行子目标执行。

### 结构化上下文

弥补长任务中的目标漂移、变量混淆和约束遗失：

- TaskContract；
- WorkUnit；
- Claim-Evidence 图；
- Artifact Store；
- 路线、风险和验证状态；
- 可恢复的 Context Pack。

所有 Agent 共享同一个任务空间，但不共享全部 token。

### 工具调用

弥补模型缺少的确定性能力和外部信息：

- Python、SymPy 和数值计算；
- pytest、静态分析和代码沙箱；
- 搜索、网页读取和引用检查；
- 文档解析和结构检查；
- Lean 或其他形式化验证；
- 数据库和专业仿真。

模型负责提出步骤和解释结果，工具负责执行、测量和生成凭证。

## 理想推理闭环

```text
理解任务
→ 编译任务契约
→ 拆解问题
→ 生成候选路线
→ 调用工具获得证据
→ 独立检查
→ 根据 blocker 修复
→ 更新结构化上下文
→ 判断继续或停止
→ 输出已验证结论
```

## 系统能力公式

```text
System Reasoning Capability
=
Base Model Capability
+ Search and Decomposition
+ Persistent Structured Context
+ Tool-grounded Evidence
+ Independent Verification
+ Iterative Repair
- Coordination Overhead
- Context Distortion
- Routing Errors
```

项目的核心工程任务，是让正向收益稳定超过协调、上下文失真和路由误判成本。

## 能力补偿的边界

框架适合弥补：

- 容易遗漏但可被检查的约束；
- 可通过工具复算的计算错误；
- 可通过独立审查发现的逻辑跳步；
- 长任务中的状态遗失；
- 需要外部知识、执行环境或资料检索的问题；
- 可以根据明确 blocker 修复的局部错误。

框架较难弥补：

- 基础模型根本无法理解的问题；
- 需要大量隐含世界模型而工具无法补充的问题；
- 无法定义成功标准和验证依据的开放任务；
- 模型无法稳定遵守输出协议；
- 错误来源无法被任何 verifier 观察到的任务。

## 倒 U 形收益假设

OpenO1 的相对增益可能与基础模型能力呈倒 U 形：

```text
极弱模型：协调成本 > 框架收益
中等能力本地模型：框架收益最大
强模型：绝对质量最高，相对增益缩小
```

原因：

- 极弱模型难以理解复杂协议、工具反馈和修复指令；
- 中等模型具备执行框架的能力，同时仍有大量可验证、可修复错误；
- 顶级模型自身已经完成较多规划和自检，框架仍有收益，但相对提升可能较小。

这一假设需要通过不同模型规模和能力档位的实验验证。

## 研究问题

1. 多大的模型开始能够稳定执行 OpenO1 协议？
2. 哪些错误类型最适合通过 Agent、上下文或工具补偿？
3. 同等计算预算下，更强单模型与多次较弱模型调用哪个更有效？
4. 多 Agent 增益中，有多少来自独立信息，有多少只是增加采样次数？
5. 工具验证能拦截多少错误，又会误杀多少正确结果？
6. 上下文结构化能减少多少目标漂移和约束遗漏？
7. 框架收益是否随模型能力呈倒 U 形？

## 实验矩阵

### 模型维度

选择多个能力层级：

```text
4B 左右
8B / 9B 左右
12B / 14B 左右
27B / 32B 左右
更强 API 或大型本地模型作为上界
```

### 工作流维度

```text
直接回答
单次回答 + 自检
Producer + 独立 Evaluator
Producer + 工具验证
Producer + Evaluator + 修复循环
多路线 Producer + 验证 + 修复
OpenO1 完整闭环
```

### 指标

- 原始正确率；
- 验证后正确率；
- 错误拦截率；
- 正确答案误杀率；
- 修复成功率；
- 目标或约束遗漏率；
- token、延迟和工具调用成本；
- 每增加 1% 正确率的额外成本；
- 协调失败、路由失败和上下文失真比例。

## 第一阶段任务选择

数学适合作为首个验证场景：

- 成功标准明确；
- 正确率容易测量；
- 错误类型可分类；
- 可以接入数值、符号和形式化工具；
- 适合比较直接回答、自检、独立验证和修复循环。

但协议从一开始应支持代码、搜索、文档和科学分析，避免数学专用设计固化到核心引擎。

## 项目表述建议

谨慎表述：

> OpenO1 研究不同能力层级的模型在可验证推理框架中能够获得多少增益，以及 Agent、结构化上下文和工具调用对哪些推理缺陷具有可测量的补偿作用。

避免未经实验直接宣称：

```text
任何小模型接入后都能达到 o1
多 Agent 必然优于单 Agent
工具越多能力越强
完整工作流一定优于增加模型参数
```

## 后续行动

1. 将系统级推理能力公式和倒 U 形假设加入项目研究路线。
2. 构建按模型规模和工作流深度分层的 benchmark 矩阵。
3. 每个模块都设计消融实验，证明独立贡献。
4. 同时测量正确率和协调成本，避免只报告能力增益。
5. 以数学为首个实验场景，但保持核心协议领域无关。
