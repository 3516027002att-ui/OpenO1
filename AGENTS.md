## 1. 项目愿景

OpenO1 是推理特化 agent 项目。

终极目标：
- 构建模型无关的推理增强 agent。
- 让任意接入 OpenO1 的模型，在推理任务中尽可能呈现类似 GPT Pro 或 Gemini Deep Think 的 heavy model 行为。
- 通过 AgentTeam、共享上下文、自动推导检查、质量审阅和继续推理判断，把单模型输出提升为可审阅、可复查、可迭代的推理过程。

短期目标：
- 从数学推理开始。
- 以可本地部署的开源模型为基础。
- 通过轻量微调和固定化推理工作流，使模型在 OpenO1 上的数学 benchmark 表现逐步逼近 `gpto1` 风格能力。
- 进展必须用可复现实验和 benchmark 记录表达，不允许把主观感受写成能力结论。

非目标：
- 不承诺仅靠提示词把弱模型伪装成强模型。
- 不用关闭校验、跳过审阅、吞掉失败来制造表面成功。
- 不在没有实验记录的情况下宣称“接近 gpto1”或“达到 heavy model 能力”。
- 不把中心引擎尚未实现的能力写成已经实现。

## 2. 中心引擎规范

中心引擎是 OpenO1 的唯一任务编排者。当前阶段采用规范先行：本文件先定义中心引擎职责和边界，不绑定具体代码路径。等真实代码结构出现后，再把入口路径写回本文件。

中心引擎职责：
- 接收用户任务、模型能力、可用工具、上下文和 benchmark 配置。
- 生成推理计划，并决定需要哪些功能模块参与。
- 维护共享上下文，统一记录题目条件、符号定义、假设、已验证结论、争议点和失败原因。
- 协调 AgentTeam、数学推导检查器、审阅门禁和继续/停止判断。
- 输出最终答案、继续推理指令、检查报告或明确失败结论。

中心引擎边界：
- 子模块不得绕过中心引擎直接改写任务目标、共享结论或最终答案。
- 子模块只能提交结构化中间结论、检查结果、候选解法、反例或审阅意见。
- 最终输出必须经过 Review Gate。
- 审阅失败时，中心引擎必须选择继续推理，或明确报告失败原因和未满足条件。

逻辑接口：
- 输入：用户任务、模型能力、可用工具、共享上下文、benchmark 配置、输出约束。
- 输出：推理计划、共享上下文更新、数学检查报告、审阅结论、最终答案或继续推理指令。

## 3. 功能模块

### 3.1 Task Intake

职责：
- 解析用户题目、目标、约束、输出格式和成功标准。
- 识别任务类型：数学证明、计算题、竞赛题、代码辅助推理、benchmark 评测或项目开发。
- 标记缺失条件和歧义点。

输出：
- 标准化任务描述。
- 已知条件列表。
- 目标结论或目标数值。
- 约束和输出格式。

### 3.2 Reasoning Planner

职责：
- 拆解数学问题。
- 选择候选推导路线。
- 决定是否需要代数验证、数值验证、反例搜索、形式化检查或多 agent 交叉审阅。

输出：
- 推理计划。
- 候选路线。
- 风险点。
- 需要调用的模块。

### 3.3 AgentTeam

职责：
- 按中心引擎分派生成候选解法、替代路径、反例、边界条件分析和形式检查建议。
- 不直接给最终答案。
- 不直接覆盖共享上下文中的已验证结论。

建议角色：
- Solver：生成主解法。
- Critic：寻找漏洞、跳步和反例。
- Verifier：检查计算、代数和逻辑一致性。
- Refiner：在审阅失败后修复推导并压缩表达。

输出：
- 候选推导。
- 反例或失败说明。
- 检查意见。
- 可复查的中间结论。

### 3.4 Shared Context

职责：
- 作为所有模块共享的事实层。
- 保存题目条件、符号定义、推导状态、已验证结论、未验证假设、争议点和失败原因。

状态标签：
- `待验证`：已提出但未检查。
- `已验证`：通过当前可用检查。
- `被推翻`：发现反例、计算错误或逻辑矛盾。
- `需要复查`：部分可信，但仍有边界条件、跳步或工具限制。

约束：
- 共享上下文中的结论必须带来源模块和状态标签。
- 不允许把未验证假设写成已验证结论。
- 冲突结论必须并存记录，并交给中心引擎裁决。

### 3.5 Math Derivation Checker

职责：
- 检查代数变形、逻辑蕴含、边界条件、定义域、单位、符号一致性和计算结果。
- 对可工具验证的步骤优先使用工具或最小可复现计算。
- 对不可完全验证的步骤明确标记限制。

输出：
- 通过项。
- 失败项。
- 可疑跳步。
- 需要继续推理或补充验证的位置。

### 3.6 Review Gate

职责：
- 判断任务是否高质量完成。
- 判断是否允许输出最终答案。
- 判断是否应该继续推理。

通过条件：
- 目标已被明确回答。
- 关键推导无未处理跳步。
- 数学检查未发现阻断性错误。
- 最终答案区分已验证推导、未验证假设和可能风险。
- 输出格式符合用户要求。

失败处理：
- 若可修复，返回中心引擎继续推理。
- 若不可修复，输出失败原因、阻断点和下一步建议。
- 不允许审阅失败后直接输出伪完成结论。

### 3.7 Benchmark / Eval

职责：
- 围绕 AIME、MATH、GSM8K 等可复现题集记录数学推理表现。
- 记录模型、数据、提示/工作流版本、微调配置、运行命令、评测时间和结果摘要。
- 比较同一模型在 OpenO1 工作流前后的表现变化。

约束：
- benchmark 结论必须基于可复现实验记录。
- 不允许用少量手工样例替代正式 benchmark 结论。
- 不允许把一次失败或成功泛化为整体能力结论。

## 4. 协作协议

模块协作必须遵守以下协议：
- 中心引擎先接收任务，再分派模块。
- 每个模块只负责自己的职责边界。
- 模块输出必须是可审阅的结构化中间结论。
- 共享上下文是跨模块事实源，不是自由文本草稿区。
- 所有最终答案都必须经过 Review Gate。
- 如果用户要求快速回答，也不能跳过关键数学检查；只能缩小检查范围并明确风险。

推荐中间结论格式：
- `claim`：结论内容。
- `status`：`待验证`、`已验证`、`被推翻` 或 `需要复查`。
- `source`：产生该结论的模块。
- `evidence`：推导、计算、工具输出或审阅依据。
- `risk`：仍需注意的问题。

## 5. 数学任务质量门禁

数学任务最终输出必须满足：
- 明确题目目标和最终结论。
- 关键符号和变量定义一致。
- 推导步骤没有影响结论的隐性跳步。
- 对定义域、边界条件、特殊值和等价变形进行必要检查。
- 计算题给出可复查的计算路径。
- 证明题给出逻辑链条，不用直觉替代证明。
- 若仍有未验证假设，必须显式标注。

严禁：
- 把“看起来合理”当成证明。
- 把未经检查的候选答案当成最终答案。
- 为了缩短输出隐藏关键推导错误。
- 用 benchmark 外的零散样例宣称整体能力。
