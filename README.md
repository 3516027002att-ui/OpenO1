# OpenO1

OpenO1 想研究一个很直接的问题：如果给模型足够多的推理时间、调用次数和计算资源，怎样把这些额外计算真正转化为更强的问题求解能力？

单次回答很难处理需要长期推导、反复实验和多路线验证的问题。OpenO1 试图用外层运行时把多轮模型调用、搜索、代码执行、符号演算、并行探索和外部验证组织起来，让系统能在同一个问题上持续推进，并保留已经得到的证据、失败路线和未解决问题。

项目目前处于早期阶段，先从数学推理开始验证。当前工作包括：

- 维护可恢复的任务与推理状态
- 在不同解法之间分支、回溯和重新分配计算
- 接入代码、符号计算和领域验证器
- 记录每次调用、工具结果、失败原因和最终证据
- 比较直接回答与长时间运行后的准确率、稳定性和计算开销

## 设计取向

OpenO1 不预设固定的 Solver-Critic-Judge 或多 Agent 流程。任务简单时，系统可以直接求解；问题复杂时，再增加分支、工具、审阅和验证。多一次调用必须带来新的推导、证据、反例或状态变化，否则只是浪费计算。

运行时负责保存状态、执行动作、控制分支和批准结束。模型可以提出结论和下一步，但关键结论需要经过验证，失败的路线也会保留下来，避免系统反复走进同一条死路。

## 当前目标

第一阶段希望回答一个可测的问题：随着推理预算增加，OpenO1 能否在一段足够长的区间内持续提高困难问题的求解质量，而不是迅速饱和、重复或退化。

Turn more compute into more intelligence.
